Enhance the Visualisation & Dashboard page using the data the tool already collects, without changing the current aesthetic or layout principles.

Keep: top summary cards, module table, existing radar chart.

Add: a small number of programme-level visuals that show patterns across the programme (by stage, modality, equity, resources, staff readiness, assessment types).

Use: the existing charting/layout system and whatever data structures are already in place (e.g. module list, stage, recommended modality, scoring breakdown, panel inputs).

The goal is to answer good curriculum questions, not to hard-wire a particular implementation.

General design requirements

Reuse the current design language

Same card style, typography, colours, hover effects, and spacing.

New visuals should look like they “always belonged” on this page.

Respect current code structure

Use existing programme/module data models and selectors.

Derive aggregates from whatever module-level data is already available (stage, credits, recommended modality, risk, panel inputs), rather than introducing new storage.

Programme-level, not module-level

Everything added here should be summaries or patterns across the modules in the selected programme.

Lightweight & performant

Avoid heavy computations on every render.

Aggregate once per programme selection (e.g. memoised or computed in a helper).

Readable at a glance

No more than 2–3 charts per row on desktop; stack vertically on smaller screens.

Keep legends and axis labels concise.

Visual 1: Modality mix by stage

Question it answers:
“How are delivery modalities distributed across the stages of this programme?”

High-level behaviour:

For the selected programme, group modules by stage and summarise how many are In-Person / Blended / Online / HyFlex.

Present this as a stacked bar (or similar) so teams can see, per stage, the proportion of each modality.

Use the same colours as the modality badges already used in the UI.

Visual 2: Weekly contact hours by stage

Question it answers:
“Where is the contact load concentrated across the programme stages?”

High-level behaviour:

For each stage, estimate or use stored contact hours per week across all modules.

Present as a simple bar chart: Stage on the x-axis, hours on the y-axis.

This visual is about relative balance; it does not need perfect precision, only a reasonable, consistent estimate based on the existing data model.

Visual 3: Equity profile by stage

Question it answers:
“Where might learners with commuting, work, or access constraints be under pressure?”

High-level behaviour:

Use the Learner Profile & Equity inputs already stored for modules (e.g. commuters, working learners, digital access).

At programme level, summarise by stage how often those flags occur.

Present this as either:

grouped bars per stage, or

a small heatmap-style grid.

The aim is to highlight stages where high on-campus expectations coincide with commuter or working-learner profiles.

Visual 4: Specialist resource demand (labs, studios, HyFlex rooms)

Question it answers:
“What specialist resource demands does this programme create?”

High-level behaviour:

Use the Resources & Constraints inputs (lab required, studio required, specialist equipment, HyFlex room etc.).

Summarise how many modules in the programme indicate each type of specialist requirement.

Present as a horizontal bar chart or a simple table with bars, showing relative demand for each resource type.

Visual 5: Programme alignment profile (5-dimension radar)

Question it answers:
“What is the overall ‘shape’ of this programme across the five scoring dimensions?”

High-level behaviour:

Reuse the five dimension scores already calculated per module (Learning Design, Assessment Fit, Learner Profile & Equity, Stage Fit, Feasibility).

Compute an average per dimension across all modules in the selected programme.

Display a single radar chart with these five averaged values.

Optionally include a subtle reference “target” (e.g. a faint ring at 80) for context, but keep it visually simple.

Visual 6: Staff digital readiness (module-level view)

Question it answers:
“What is the overall staff digital confidence and online experience picture for this programme?”

High-level behaviour:

Use the Staff Profile inputs already captured at module level (digital confidence, online experience).

Summarise the proportions of modules associated with low / moderate / high digital confidence and with none / some / substantial online experience.

Present this as one or two small pie/doughnut charts or equivalent, inside a single card.

Add short descriptive text clarifying that this is for planning support, not evaluation.

Visual 7: Assessment–modality pattern (major assessments)

Question it answers:
“How are our major assessment types distributed across modalities?”

High-level behaviour:

From the Assessment Types inputs, identify which assessment types are marked as “major” in each module.

Cross-tabulate major assessment types against recommended modality.

Display this as a matrix-like table with subtle cell shading (heatmap effect) to indicate where certain assessment types cluster (e.g. many exams in in-person modules, portfolios mostly online/blended).

This should be readable as a simple table with counts, even without colour.

Interaction / filtering

All new visuals should respond to the programme selector already on the page.

If at any point there is “no data” for a particular aggregate (e.g. no staff profile entered yet), show a short neutral message in that card rather than an empty or broken chart.
